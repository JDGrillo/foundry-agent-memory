## Description
Implement the user simulator module that provides scripted user messages for each scenario. Each scenario requires multi-turn conversations with consistent user IDs across sessions to demonstrate memory continuity.

## Acceptance Criteria
- [ ] AC-1: `src/common/user_simulator.py` provides conversation scripts for each scenario
- [ ] AC-2: Help Desk: messages for user "user_123" across two sessions (first contact + return visit)
- [ ] AC-3: Standup: messages for 2-3 team members (e.g., Alice, Bob) across two days
- [ ] AC-4: Customer Success: messages for CSM-1 "Sarah" and CSM-2 "Mike" about account "Acme Corp"
- [ ] AC-5: User IDs are consistent across sessions (same user_id for return visits)
- [ ] AC-6: All simulated data uses obviously fake names/details (REQ-NF-006)

## Technical Notes
- Simple data structure: list of dicts or dataclass with `user_id`, `session`, `messages[]`
- Messages should be realistic enough to trigger meaningful agent responses:
  - Help Desk Session 1: "My laptop keeps crashing" → provides device info when asked → gets resolution
  - Help Desk Session 2: "Screen flickering again" (related issue, expects agent to remember device)
  - Standup Day 1: Alice working on auth refactor (blocked by API team), Bob finishing DB migration
  - Standup Day 2: Alice still blocked, Bob done with DB + picking up frontend
  - CS: Sarah logs interaction with Acme Corp (contract renewal, feature request); Mike picks up Acme account
- Keep message content concise but information-rich
