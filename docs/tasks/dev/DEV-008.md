## Description
Implement the Project Standup Bot scenario demonstrating group/team memory. The script provisions the `standup-memory` store with a static team scope, creates a portal-visible agent, and runs two simulated standups:
1. Day 1 — team members report status, blockers, and plans; all stored under shared team scope
2. Day 2 — agent recalls yesterday's context, follows up on blockers, notices patterns

The agent and all threads must be visible in the Azure AI Foundry portal.

## Acceptance Criteria
- [ ] AC-1: `src/scenarios/standup.py` is independently runnable via `python -m src.scenarios.standup`
- [ ] AC-2: Script provisions `standup-memory` store (idempotent) with team scope config
- [ ] AC-3: Script creates Standup Bot agent via Foundry Agents Service (portal-visible)
- [ ] AC-4: Day 1: simulates 2-3 team members providing standup updates (work, blockers, next steps)
- [ ] AC-5: Agent acknowledges and summarizes each update
- [ ] AC-6: Memory store contains all updates under `team-alpha` scope
- [ ] AC-7: Day 2: agent recalls yesterday's blockers and asks for updates on them
- [ ] AC-8: Agent notices persistent blockers and suggests escalation
- [ ] AC-9: Agent tracks progress (e.g., "Bob finished DB migration, now on frontend")
- [ ] AC-10: Console shows retrieved team context before new standup begins
- [ ] AC-11: (P1) After Day 2, generates structured summary: recurring blockers, completed items, items in progress >1 day
- [ ] AC-12: Agent and threads visible in Foundry portal after execution

## Technical Notes
- All team members interact via the same thread scope (`team-alpha`) so memory is shared
- Each team member's messages should be distinguishable (include name in message or use metadata)
- Day 2 thread is new but agent retrieves from same scope, demonstrating group memory persistence
- The summary generation (AC-11) can be a final agent prompt: "Generate a standup summary highlighting patterns"
- Use console formatter (DEV-006) to clearly separate team members' turns
